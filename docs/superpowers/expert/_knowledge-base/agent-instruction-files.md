## Updated 2026-06-30 — AGENTS.md/CLAUDE.md layering + standard status (v:onboard audit)

### AGENTS.md governance status (verified)
- AGENTS.md was released by **OpenAI in Aug 2025**, contributed to the **Linux Foundation Agentic AI Foundation (AAIF)** formed **Dec 2025** alongside MCP (Anthropic) and goose (Block). 60k+ repos adopted. So "Linux Foundation AAIF standard" in the spec is **accurate**. Sources: [LF press](https://www.linuxfoundation.org/press/linux-foundation-announces-the-formation-of-the-agentic-ai-foundation), [OpenAI](https://openai.com/index/agentic-ai-foundation/), [agents.md].
- Caveat: AGENTS.md is a **convention, not a strict schema** — no enforced section list, no precedence spec across tools. Each consumer (Codex, Cursor, Copilot, Gemini, Claude-via-import) interprets it.

### Claude Code memory model (verified against code.claude.com/docs/en/memory, 2026-06-30)
- **Claude Code does NOT natively read AGENTS.md.** Exact doc text: "Claude Code reads `CLAUDE.md`, not `AGENTS.md`." Bridge via `@AGENTS.md` import or symlink. → A repo that ships ONLY AGENTS.md is invisible to Claude Code until a CLAUDE.md bridge exists. This is the load-bearing fact behind the thin-bridge design.
- **Load order (broad→specific), all CONCATENATED (no override):** Managed policy → User (`~/.claude/CLAUDE.md`) → Project (`./CLAUDE.md` or `./.claude/CLAUDE.md`) → Local (`./CLAUDE.local.md`). Within a dir, `CLAUDE.local.md` appends after `CLAUDE.md`. Up the tree, root→cwd, closer-to-cwd read LAST.
- **Precedence on conflict is positional, not semantic.** Files are concatenated; "if two rules contradict, Claude may pick one arbitrarily." There is no hard "AGENTS.md wins" or "CLAUDE.md wins" rule inside Claude Code — community claims of strict precedence are folklore, not doc-backed. In an `@AGENTS.md` + below-the-import Claude section, the Claude-specific lines appear AFTER the import → tend to be honored later, but this is soft.
- **`@import` loads in full at launch; not a token optimization.** Doc: "imported files still load and enter the context window at launch." Max import depth 4 hops. First-time external imports show an approval dialog; declining disables silently.
- **Size: "target under 200 lines per CLAUDE.md file." Soft target, not a hard ceiling** — "CLAUDE.md files are loaded in full regardless of length, though shorter files produce better adherence." (The 200-line/25KB cap is on auto-memory `MEMORY.md`, NOT CLAUDE.md.) Spec's "200-line ceiling" overstates: it's a recommendation.
- **Managed-policy CLAUDE.md cannot be excluded** by any user/project/local setting (`claudeMdExcludes` is ignored for it). Org security/compliance text always loads. → onboarding must NEVER assume it can override or restructure managed-layer content; it can only add at project layer.
- **`/init` already does detect-and-bridge natively.** Doc: "Running `/init` in a repo that already has an `AGENTS.md` reads it and incorporates the relevant parts… It also reads other tool configs like `.cursorrules`, `.devin/rules/`, and `.windsurfrules`." `CLAUDE_CODE_NEW_INIT=1` adds an interactive explore→propose→review-before-write flow. → /v:onboard's reconciliation overlaps native /init; the differentiator must be the citation/recall loop, not the bridging itself.
- **CLAUDE.md is context, not enforcement.** Doc: "To block an action regardless of what Claude decides, use a PreToolUse hook." Validates the spec's "aspirational rules → hooks" diagnosis.

### Practitioner pain (Layer 3, dated)
- Symlink CLAUDE.md→AGENTS.md "often gets confused, requiring several iterations" — HN [45788866](https://news.ycombinator.com/item?id=45788866) (2025-11). Multiple HN comments report agents inconsistently reading AGENTS.md ([45791391](https://news.ycombinator.com/item?id=45791391)). Isolated-to-moderate signal; not a measured rate.
- Nested/monorepo AGENTS.md: Codex concatenates global + every AGENTS.md root→cwd, **capped 32 KiB**, closer = higher precedence; cap "reached quickly" with 3 nested files; unscoped nested files leak wrong-domain rules. Sources: [codegateway 2026](https://www.codegateway.dev/en/blog/agents-md-playbook-2026), [danielvaughan 2026-03](https://codex.danielvaughan.com/2026/03/26/agents-md-advanced-patterns/), [agents.md issue #53](https://github.com/agentsmd/agents.md/issues/53).
- `/init`-generated CLAUDE.md "often hallucinates file structures, recommends wrong libraries, and misses the coding style" — [obviousworks 2026](https://www.obviousworks.ch/en/designing-claude-md-right-the-2026-architecture-that-finally-makes-claude-code-work/). This is the exact failure /v:onboard's citation gate must beat — and the reason auto-generated architecture prose is distrusted by default.

### MCP security — lethal trifecta (verified, primary sources)
- **Lethal trifecta** = private data + untrusted content + external communication, coined Simon Willison Jun 2025: [simonwillison.net/2025/Jun/16](https://simonwillison.net/2025/Jun/16/the-lethal-trifecta/). Willison's stance: "the only way to stay safe… is to avoid that lethal trifecta combination entirely"; guardrail/scanner mitigations are insufficient because "in application security, 99% is a failing grade."
- **Supabase service-role incident is REAL** (disclosed Jul 2025): support-ticket prompt injection → Cursor agent with `service_role` (bypasses RLS) reads `integration_tokens`, writes to `support_messages` where attacker reads it. Fix that kills the channel: `--read-only`. Sources: [Simon Willison Jul 6 2025](https://simonwillison.net/2025/Jul/6/supabase-mcp-lethal-trifecta/), [General Analysis](https://generalanalysis.com/blog/supabase-mcp-blog), [Supabase defense-in-depth](https://supabase.com/blog/defense-in-depth-mcp).
- **GitHub toxic-agent-flow incident is REAL** (Invariant Labs, May 2025): poisoned public issue → agent with broad PAT reads PRIVATE repos → writes stolen content into a public PR. Root cause "architectural, not a bug in GitHub's server code." Mitigations: one-repo-per-session, least-privilege PAT. Sources: [Invariant Labs](https://invariantlabs.ai/blog/mcp-github-vulnerability), [Invariant toxic-flow](https://invariantlabs.ai/blog/toxic-flow-analysis), [github-mcp-server#844](https://github.com/github/github-mcp-server/issues/844), [devclass](https://devclass.com/2025/05/27/researchers-warn-of-prompt-injection-vulnerability-in-github-mcp-with-no-obvious-fix/).
- **Key nuance for warn-vs-block:** in BOTH incidents the write capability is what closes the exfiltration loop. `--read-only` / least-privilege provably neutralizes them. So a recommender that DEFAULTS to read-only flags (spec already does this for Postgres/Supabase) has mostly defused the trifecta before the warning even fires. The residual danger is the GitHub-PAT-scope case: read-only does NOT help (it's a read→read→write-to-public-PR chain) — only single-repo session scoping / token-scoping does.
