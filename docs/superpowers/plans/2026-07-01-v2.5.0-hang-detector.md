# Hang-detector (v2.5.0) Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: superpowers:test-driven-development for Task 1. Steps use checkbox (`- [ ]`) tracking.

**Goal:** Detect stuck/parked orchestrator jobs (liveness probe from git+FS), surface them in `/v:status`, auto-remediate in the dispatcher, and close the one uncapped external-launch path.

**Architecture:** A stdlib probe reads `state.json` and classifies each `running` job from **git + filesystem** signals only. `/v:status` renders it; the dispatcher runs it at batch boundaries (auto-collect `LIKELY-DONE`, surface/kill `STALE`). Reuses the existing `failure-policy` + timeout supervisor — no new retry mechanism, no daemon.

**Tech Stack:** Python 3.9 stdlib, Markdown agent/command/skill docs, bash worker script, JSON config.

## Global Constraints
- **Never model-self-reported** — every signal is git / filesystem (mtime). Same ethos as the scope gate.
- **Degrade-safe** — any missing/unreadable signal ⇒ `UNKNOWN`; the probe never crashes `/v:status` or the dispatch loop.
- **No daemon, no new deps** (stdlib only). One-shot CLI called on demand.
- **Reuse existing budgets** — STALE/DEAD remediation flows through `failure-policy.md` (`timeout` class → retry, `max_total_retries`), not a new path.
- **Anti-over-engineering (from archaeology):** the 3 worker scripts already enforce supervisor+`</dev/null`; external workers run synchronously under the hard cap. So the probe does **not** police external workers (the supervisor does) and does **not** add `pid` bookkeeping. Its unique value is **Claude subagent** `LIKELY-DONE`/`STALE`.
- **Version lockstep:** `plugin.json` AND `marketplace.json` → `2.5.0` (CI enforces).

### Task 1: Liveness probe (serial spine)
**Files:** Create `scripts/compound-v-liveness.py`

**Interfaces (Produces):**
- CLI: `compound-v-liveness.py <run-dir> [--stale-sec N] [--json]` → per-running-job classification; `--selftest`.
- `classify_job(job: dict, now: float, stale_sec: int) -> (liveness, evidence, last_progress_s)`; `liveness ∈ {WORKING, LIKELY-DONE, STALE, DEAD, UNKNOWN}`.
- `probe(run_dir, stale_sec, now) -> {job_id: {liveness, evidence, last_progress_s}}`.
- Exit: `0` if none STALE/DEAD (LIKELY-DONE is exit 0 — good news); `3` if any STALE/DEAD; `2` usage error.

**Classification (git+FS only):**
1. `LIKELY-DONE` — worktree exists and `git -C <wt> rev-parse HEAD != baseline` (a commit landed; only the notification is stuck). Checked FIRST.
2. mtime signal — newest mtime of working-tree files under the worktree (**excluding `.git`**), and an optional `log` path: `age ≤ stale_sec` ⇒ `WORKING`; `age > stale_sec` ⇒ `STALE` (or `DEAD` if a `pid` is present and not alive).
3. No worktree/mtime signal — `pid` present: alive ⇒ `UNKNOWN` (running, no FS signal), dead ⇒ `DEAD`; no pid ⇒ `UNKNOWN`.
- `.git` is excluded from the mtime walk so a `git init`/commit doesn't mask staleness (commits are caught by rule 1).

- [ ] Step 1: Write `compound-v-liveness.py` with the full implementation + a `--selftest` that builds a tempdir fixture per class (real `git init` for LIKELY-DONE: commit=baseline, second commit=HEAD; `os.utime` old mtime for STALE; fresh file for WORKING; dead pid for DEAD; empty job for UNKNOWN), mirroring the `check(name, cond)` pattern in `compound-v-run-with-timeout.py:_selftest`.
- [ ] Step 2: `python3 scripts/compound-v-liveness.py --selftest` → all classes pass.
- [ ] Step 3: `chmod +x`; smoke on a real completed run dir (all jobs `done` ⇒ empty result, exit 0).
- [ ] Step 4: Commit.
