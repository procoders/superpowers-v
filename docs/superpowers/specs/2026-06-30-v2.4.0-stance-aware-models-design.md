# Stance-aware models map — cost-aware routes Claude `standard` to Sonnet 5 (design, v2.4.0)

**Status:** approved for build (2026-06-30). Scope confirmed: in the `cost-aware` stance, the
entire Claude `standard` tier (including `core_slice` and `tests_new`) resolves to Sonnet 5.

## 1. What it is
Make tier→model resolution **stance-aware** so the `cost-aware` stance routes Claude `standard`-tier
implementers to **Sonnet 5** (`claude-sonnet-5`, ≈ Opus 4.8 at lower cost), while `balanced` (the
default) keeps `standard → opus`. The `cost-aware` stance already exists in
[`routing-policy.md`](../../skills/compound-v/routing-policy.md) and differs from `balanced` only in
**tier assignment**; today the `tier → model` map is **stance-independent**, so cost-aware's
`standard` still resolves to Opus. `routing-policy.md` already calls it "the **Balanced** models map"
— per-stance maps were always the intent; this realizes them.

## 2. Why
Sonnet 5 (released 2026-06-30) benchmarks close to Opus 4.8 at lower cost. For well-specified,
lower-risk work, the `cost-aware` stance should route standard-tier implementers to it. This is an
**opt-in routing change keyed on the stance**, NOT a default shift — `balanced` stays Opus-by-default.

## 3. The change (one value cell + the mechanism around it)
Per-stance built-in Claude maps:

| stance | claude `{deep, standard, light}` | changed? |
|---|---|---|
| `balanced` | `{opus, opus, sonnet}` | no |
| `conservative` | `{opus, opus, sonnet}` | no |
| **`cost-aware`** | **`{opus, sonnet, sonnet}`** | ✅ the only value change |
| `claude-only` | `{opus, opus, sonnet}` | no |

`codex` / `antigravity` / `cursor` maps are **identical across stances** (current values). The
`"sonnet"` alias resolves to Sonnet 5 via Claude Code's native tier alias — no concrete ID is pinned.

## 4. Components

### 4.1 Resolver — `scripts/compound-v-resolve-model.py`
- `DEFAULT_MODELS` → `DEFAULT_MODELS_BY_STANCE = {<stance>: {<backend>: {<tier>: model}}}` for all four
  stances (`balanced`, `conservative`, `cost-aware`, `claude-only`).
- New `--stance` arg, `∈ {balanced, conservative, cost-aware, claude-only}`, **default `balanced`**.
- `resolve(backend, tier, stance, effort, config_models, explicit_model)`: select the stance's map,
  then `[backend][tier]`. Precedence is unchanged: `explicit_model` > config override > stance
  built-in default.
- Because the default is `balanced`, every existing caller that does not pass `--stance` keeps the
  current behavior (`standard → opus`) — fully backward-compatible.

### 4.2 Config backward-compat — `.claude/compound-v.json` `models`
Two shapes are accepted, detected by whether the top-level keys are stance names:
- **Legacy flat** — `models: {<backend>: {<tier>: model}}` — applied to **all** stances (a global
  per-cell override). Existing seeded configs keep working unchanged.
- **Per-stance** — `models: {<stance>: {<backend>: {<tier>: model}}}` — overrides a single stance's
  cell. `/v:init` seeds this shape going forward.

### 4.3 Dispatcher passes `--stance`
`parallel-dispatcher` / `adapter-claude` / `phase-3-parallel-opus-dispatch` resolve each job's model
with `--stance <manifest.routing_stance>`. A manifest already carries `routing_stance`; the dispatcher
threads it into the resolver call. No flag ⇒ `balanced`.

### 4.4 partition-reviewer stance-awareness
The reviewer reads the manifest's `routing_stance`. Under `cost-aware`, a `standard`-tier Claude job
resolving to Sonnet is the **policy default** — so the `SONNET_INELIGIBLE` / `SONNET_UNJUSTIFIED`
8-box justification check does **not** apply to it. Under `balanced` / `conservative` the check
applies exactly as today (`standard → opus`, so Sonnet appears only on `light` = the junior-task
tier). The deterministic validator (`compound-v-validate-manifest.py`) is unchanged — it never
adjudicated implementer Sonnet eligibility (that is the reviewer's judgment).

## 5. Invariants (hold in every stance)
1. **Reviewers ⇒ deep ⇒ opus** (validator + partition-reviewer) — stance-independent.
2. **Security / auth / payments / PII / a11y ⇒ deep ⇒ opus** in every stance — sensitive surfaces are
   never cost-optimized.
3. **Never Haiku**, anywhere (map, override, frontmatter).
4. **`balanced` (default) is unchanged** — `standard → opus`. Only `cost-aware.claude.standard`
   changes (`opus → sonnet`).
5. The resolver still **fails closed** (non-zero) when a `(stance, backend, tier)` cannot resolve.

## 7. Verification
- **Resolver `--selftest`:** `cost-aware` claude `standard` → `sonnet` AND `balanced` claude `standard`
  → `opus`; default stance (`--stance` omitted) == `balanced`; legacy flat config applies across all
  stances; a per-stance config overrides exactly one stance's cell; an unknown stance errors.
- **partition-reviewer:** a `cost-aware` manifest with a `standard`-tier Claude (Sonnet) `core_slice` /
  `tests_new` job is **NOT** flagged `SONNET_UNJUSTIFIED`; the same job under `balanced` **is** flagged.
- **Dogfood `/v:dispatch`:** materialize/dispatch a small change with `routing_stance: cost-aware` and
  confirm standard-tier Claude jobs resolve to `sonnet` while reviewers + sensitive jobs stay `opus`.
- **Codex cross-model verification** of the resolver's stance logic and the partition-reviewer's
  stance handling (the maintainer's standing request — the model that writes ≠ the model that checks).

## 8. Out of scope (v2.4.0)
- Per-stance `codex` / `antigravity` / `cursor` model differences (kept identical for now).
- Concrete model-ID pinning (the `"sonnet"` alias auto-tracks Sonnet 5; pinning is a separate change
  with Task-param-compat considerations).
- Changing which job-types route to `standard` in cost-aware (the routing table is unchanged —
  `core_slice` + `tests_new` already route to `standard`).
- Any change to `balanced` / the global default — Opus-by-default is explicitly preserved.
