# Hang-detector for the Compound V orchestrator (design, v2.5.0)

**Status:** approved for build (2026-07-01). Full scope chosen by the maintainer.

## 1. What it is
A **liveness probe** over an orchestrator run's in-flight jobs, **auto-remediation** in the dispatcher, and **enforced hard-timeout wrapping** of every external worker. It turns silent forever-waits into detected, acted-upon states. All signals are **git + filesystem + PID** — never model-self-reported (same ethos as the scope gate).

## 2. Why
Two hang modes bit during the v2.4.0 dogfood (2026-07-01):
1. A headless `codex exec` ran **44 minutes hung at 0% CPU**, blocked reading stdin, because the backend-launcher adapter *describes* the timeout supervisor but does not *enforce* it — a bare `codex`/`cursor`/`agy` call has no cap.
2. Claude subagents "**park**": a worker finishes and commits in its worktree, but its completion notification never routes back, so the dispatcher waits forever (the `task-1-reviewer-gate` case — the human had to inspect the worktree and nudge the dispatcher by hand).

Neither is currently detectable. The orchestrator has no notion of "is this job still making progress?"

## 3. Components

### 3.1 Enforced external-worker launch (adapters + SKILL)
Every external worker (`codex` / `cursor` / `antigravity`) MUST launch with **`stdin </dev/null`** AND through **`scripts/compound-v-run-with-timeout.py`** (`--timeout` / `--grace` — NOT `--timeout-sec`). Codify the canonical invocation + the invariant **"a bare external-worker call is a bug"** in `skills/backend-launcher/adapter-codex.md`, `adapter-cursor.md`, `adapter-antigravity.md`, and `skills/backend-launcher/SKILL.md`. This is the root-cause fix for hang-mode #1.

### 3.2 Liveness probe — `scripts/compound-v-liveness.py <run-dir>`
Stdlib-only. Reads `state.json`; for each job with `status: running`, classifies from **objective signals only**:

| Class | Rule |
|---|---|
| `LIKELY-DONE` | the job's worktree HEAD is a commit **past its recorded `baseline`** (git), but status is still `running` → work is done, only the notification is stuck |
| `DEAD` | the job's recorded `pid` is **not alive** (`kill(pid,0)`) AND there is no commit/result |
| `STALE` | alive, but the **newest mtime** under the worktree (or the external log) is older than `stale_threshold` → no progress |
| `WORKING` | a progress signal (worktree/log mtime) advanced within `stale_threshold` |
| `UNKNOWN` | required signal absent/garbled (no worktree path, no pid, unreadable) → never crash |

Output: JSON per job `{job, liveness, elapsed_s, last_progress_s, evidence}`. **Exit ≠ 0** if any `STALE`/`DEAD`. Signals are derived from what already exists — worktree paths and (for external jobs) pids in `state.json`, plus filesystem mtimes; **no new bookkeeping is required for the probe to function** (backward-compatible with existing runs). `stale_threshold` defaults per tier (deep 900 s, else 600 s), overridable via `--stale-sec`.

### 3.3 `/v:status` — `Liveness` column
`/v:status` renders each running job's class + a one-line hint (`LIKELY-DONE → /v:resume`, `STALE → suspected hang`, `DEAD → will retry`). Read-only; degrade-safe (probe error ⇒ column shows `—`, never breaks the table).

### 3.4 Dispatcher liveness sweep + auto-remediation
Between batches — and as a heartbeat while waiting on a batch — the dispatcher runs the probe and acts:
- `LIKELY-DONE` → **collect** (scope-gate + merge + mark done). Kills the "nudge the dispatcher by hand" pain.
- `STALE` / `DEAD` → **kill** the external process group (if a pid is recorded) and **retry** via the existing `failure-policy` (respecting `max_total_retries` + the circuit breaker — no retry storm).
- A Claude subagent that is `STALE` (not an external process we can kill) is surfaced, not force-killed (harness-owned); its committed work, if any, is caught as `LIKELY-DONE`.

### 3.5 `state.json` — record `pid` for external jobs
The dispatcher records the launched **`pid`** for each external (`codex`/`cursor`/`antigravity`) job so the probe can detect `DEAD` and the sweep can kill a `STALE` one. Absent `pid` ⇒ the probe degrades that job to mtime-only signals (still classifies `LIKELY-DONE`/`STALE`/`WORKING`, just not `DEAD`). No other new fields are required.

## 4. Invariants / anti-over-engineering
- **No daemon, no poller service, no new dependencies** (stdlib only). The probe is a one-shot CLI that `/v:status` and the dispatcher call on demand.
- **Never model-self-reported** — every signal is git / filesystem / PID.
- **Degrade-safe** — any missing or unreadable signal ⇒ `UNKNOWN`; the probe never crashes `/v:status` or the dispatch loop.
- **Retry respects existing budgets** — reuses `failure-policy` (`max_total_retries`, cooldowns, circuit breaker); introduces no new retry mechanism.
- **The probe adds no required state** — it works on existing runs from git+FS+PID; `pid` recording (3.5) only *enriches* it (enables `DEAD` + kill).
- The probe is a normal Python CLI (uses `os.stat` mtimes + `time.time()`), not a workflow script — no timestamp restrictions apply.

## 5. Verification
- **`compound-v-liveness.py --selftest`** with a fixture per class: a fake worktree carrying a commit past baseline ⇒ `LIKELY-DONE`; a live pid + a stale-mtime log ⇒ `STALE`; a dead pid + no commit ⇒ `DEAD`; a fresh mtime ⇒ `WORKING`; a missing worktree/pid ⇒ `UNKNOWN`.
- **Dogfood:** a deliberately-hung fake worker is classified `STALE` and the sweep kills+retries it; a committed-but-unmerged fake worktree is classified `LIKELY-DONE` and auto-collected.
- **Codex cross-model verification** of the probe classification logic and the enforced-launch adapters (the model that writes ≠ the model that checks).
- Full regression (all existing selftests + lint + the CI version-lockstep check) stays green.

## 7. Out of scope (v2.5.0)
- Killing or managing Claude subagents (harness-owned) — we only **detect** their parked/`LIKELY-DONE` state and auto-collect committed work.
- A persistent monitor daemon or cross-run dashboard.
- Progress heuristics beyond mtime + git-commit (e.g. parsing worker logs for semantic progress).
